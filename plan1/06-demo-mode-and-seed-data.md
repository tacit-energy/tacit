# Spec 06: Demo Mode & Seed Data

Two jobs:
1. **Seed data** — dummy datasets so the UI works without any LLM, and deterministic fallbacks when an agent call/parse fails.
2. **Demo Mode** — a hidden toggle that guarantees the compounding-memory loop fires perfectly on stage, with zero LLM latency or parsing risk.

---

## 1. Seed Datasets (`src/data/*`)

All seeds use **fixed ISO timestamps** (no `new Date()` / `Math.random()`), so Export/Import and re-runs are byte-stable.

### `rawSensors.ts` — Agent 1 input

- A generated list of ~2,000 `RawSensor` entries spread across a handful of systems (Chiller plant, Heat exchange loop, Distribution pumps, Return valves, AHUs, Boilers).
- Generated by a small deterministic builder (loop with index-based names like `TT-1043`, `FT-2207`), **not** random — seeded counters only. ~2,000 is enough to make Agent 1's "collapse to 5–10 nodes" job meaningful.
- Includes a clearly identifiable **"Pump P3"** sensor cluster (used by the demo scenario, §3).

### `seedTopology.ts` — fallback simplified topology

A hand-authored `TopologyData` of ~7 nodes the demo relies on:

```
Main Chiller ──▶ Heat Exchanger ──▶ Distribution Pump P3 ──▶ Return Valve V2
      │                                     │
      └──▶ Cooling Tower                    └──▶ AHU Cluster
                                            Boiler B1 ──▶ Heat Exchanger
```

- Node `pump-p3` (name "Distribution Pump P3", type `pump`) is the demo anchor.
- Statuses: most `ok`, `pump-p3` set `warning` in the demo anomaly scenario.
- Used when Agent 1 fails/omitted or in Demo Mode.

### `seedTimeSeries.ts` — fallback time-series

- One `TimeSeriesData` per seed node, ~24 hourly `TimePoint`s over the demo window (e.g. `2026-06-27T00:00Z` … `2026-06-27T23:00Z`).
- Shapes chosen so sparklines look alive: chiller temp gently oscillates; `pump-p3` power shows a conspicuous spike/anomaly pattern.
- Deterministic values (formula on index), not random.

### `seedAnomalies.ts` — fallback anomalies

- 3–4 plausible `AnomalyData` covering both confidence types so both badges are always visible:
  - One `rule_based_data` (blue): e.g. *"Return temp rising while Pump P3 active — physical inconsistency."* — high score (e.g. 94).
  - One `ai_inferred` (orange): e.g. *"Unusual chiller cycling pattern vs. baseline."* — moderate score (e.g. 71).
- **The demo anomaly** (`anomaly-pump-p3`, §3) is always present and ordered first.
- All start `status: 'open'`.

### `demoMemory.ts` — the June-15 decision

A single canonical `DecisionMemory` entry:

```ts
export const JUNE_15_DECISION: DecisionMemory = {
  id: 'mem-2026-06-15-pump-p3',
  anomalyId: 'anomaly-pump-p3-prev',
  anomalySnapshot: {
    description: 'Distribution Pump P3 drew abnormal power and was flagged as a fault.',
    relatedNodeIds: ['pump-p3'],
    confidenceType: 'rule_based_data',
  },
  humanRationale:
    'Pump P3 was deliberately overridden on June 15 due to planned maintenance. ' +
    'The abnormal reading is expected during the maintenance window — not a fault.',
  actionTaken: 'modify',
  timestamp: '2026-06-15T09:12:00Z',
};
```

## 2. Demo Mode Behavior

A boolean `settings.demoMode`, toggled from the Left Panel (subtle/hidden switch — Spec 04 §2.5). Effects:

| Aspect | Demo Mode OFF | Demo Mode ON |
|---|---|---|
| Agent calls | Real LLM via active provider; seed fallback on failure | **Bypassed** — agents return seeds synchronously |
| `decisionMemory` on init | empty (unless imported) | **preloaded with `JUNE_15_DECISION`** |
| `startAnalysis` | runs Agents 1→2 | loads `seedTopology` + `seedTimeSeries` + `seedAnomalies` immediately; sets `analysisStage='done'` |
| Anomaly ordering | by status/score | **`anomaly-pump-p3` forced first** |
| `explainAnomaly(anomaly-pump-p3)` | runs Agent 3 | returns a **deterministic recall** of `JUNE_15_DECISION` (no network) |
| Latency | network-bound | instant |

### `toggleDemoMode()` side effects (Spec 03 §3)

- ON → if `decisionMemory` is empty, seed it with `[JUNE_15_DECISION]`; preload topology/timeSeries/anomalies so the canvas isn't blank; set provider behavior to short-circuit.
- OFF → leave existing data in place (don't wipe the operator's work); just stop short-circuiting agents. (A separate `resetState()` clears everything.)

### Deterministic Agent 3 recall in Demo Mode

In `explainAnomaly`, when `demoMode` and the anomaly relates to `pump-p3`:

```ts
return {
  isRelated: true,
  pastRationale: JUNE_15_DECISION.humanRationale,
  suggestedAction:
    'Treat as expected maintenance behavior — do not dispatch a crew. ' +
    'Confirm the maintenance window and dismiss if still scheduled.',
  relatedDecisionId: JUNE_15_DECISION.id,
};
```

This guarantees the card expands with the exact June-15 rationale, on cue, regardless of LLM availability.

### The demo anomaly (`seedAnomalies.ts`)

```ts
export const DEMO_ANOMALY: AnomalyData = {
  id: 'anomaly-pump-p3',
  description:
    'Distribution Pump P3 is drawing abnormal power and downstream return ' +
    'temperature is rising — flagged as a potential fault.',
  relatedNodeIds: ['pump-p3'],
  timestamp: '2026-06-27T08:05:00Z',
  confidenceScore: 88,
  confidenceType: 'rule_based_data',
  status: 'open',
};
```

It is engineered to be an obvious match for `JUNE_15_DECISION` so Agent 3 (real or deterministic) recalls the maintenance rationale — demonstrating compounding institutional memory.

## 3. The Stage Demo Flow

**Path A — Demo Mode (bulletproof, no network):**
1. Toggle **Demo Mode ON**. `decisionMemory` is preloaded with June-15.
2. Click **Start Analysis** → topology + sparklines render instantly; `anomaly-pump-p3` is first, blue "Rule-based" badge.
3. Click **Explain (AI)** on it → card expands instantly: *"Related to previous incident. Past human rationale: Pump P3 was deliberately overridden on June 15 due to planned maintenance…"*
4. Click **Dismiss Alert** → rationale required → type "Confirmed maintenance window still active" → commit. KPIs update.
5. Narrative: *"The AI didn't just detect — it remembered why a human overrode this two weeks ago."*

**Path B — Live LLM (the real thing), using Import/Export:**
1. Demo Mode OFF, provider = Claude with a key. Import `demo/day-1.json` (fresh anomalies, short memory).
2. Walk the loop on a `pump-p3`-style anomaly → **Modify** with a maintenance rationale → it lands in `decisionMemory`. **Export** state.
3. Import `demo/day-14.json` (the same anomaly recurs; memory now contains the override).
4. Click **Explain (AI)** → real Agent 3 recalls the rationale from memory → contextualized suggestion + **orange "Inferred from Past Decision"** flavor in the recall block.

Path A is the safety net; Path B is the headline. Both are wired off the same store and provider abstraction.

---

## 4. Generating the demo JSON files

`demo/day-1.json` and `demo/day-14.json` are produced by exporting from a seeded session (Spec 03 §4) and then hand-trimming:
- **day-1:** `decisionMemory: []` (or one unrelated entry), anomalies include a `pump-p3` event.
- **day-14:** `decisionMemory: [JUNE_15_DECISION]`, same recurring anomaly, fresh timestamps.

Keep them pretty-printed and committed so the pitch is reproducible on any machine.

---

## 5. Determinism Checklist

- [ ] No `Math.random()` in seed generation, layout, or id assignment used by demo paths (use index/counter-based ids; `lib/ids.ts` can use a monotonic counter for runtime-created decisions, which is fine since those aren't replayed).
- [ ] No `new Date()` in seed data — all timestamps are fixed ISO constants.
- [ ] Demo Mode never touches the network.
- [ ] Seed fallbacks are returned by reference-safe copies (clone before mutating) so re-running analysis doesn't corrupt the seed singletons.
